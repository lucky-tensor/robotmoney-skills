# Architecture Documentation Critique — Robot Money

> Scope: a critical review of the architecture- and product-shaped docs in
> `docs/`, primarily `docs/architecture-proposal-v0.md`, `docs/prd.md`,
> `docs/project-roadmap.md`, and `docs/technical/smart-contracts.md`,
> grounded against the four Solidity contracts under `contracts/`.
> Date: 2026-05-06.

This is a critique, not a sign-off. Findings are grouped by severity. Every
claim cites a file and line range so the author can confirm or push back.

---

## 1. Executive summary (the load-bearing problems)

1. **`architecture-proposal-v0.md` describes a different product than the one
   in this repo.** The proposal specifies a Rust payment daemon talking to a
   single-purpose USDC *escrow gateway* with agent registry, master batch
   confirmations, threshold accumulation, and refund/expiry. The deployed
   system (`contracts/RobotMoneyVault.sol:22`) is an ERC-4626 *yield vault*
   with adapters into Morpho/Aave/Compound. There is no escrow, no
   `escrowMicropayment`, no `confirmBatch`, no `paymentId`, no agent
   registry, no merchant allowlist, no Rust client. The proposal does not
   acknowledge this gap or position itself as a future/separate product, so
   a reader cannot tell whether it is a roadmap, a parallel system, dead
   spec, or a renaming of the current product.

2. **The PRD overstates what the on-chain protocol enforces.** `docs/prd.md`
   §5.1–§5.2 describes governance-set bucket weights, weekly/monthly votes,
   bucket-B/C constituent management, slippage/quote-deadline enforcement,
   and atomic vault+basket execution as protocol features.
   `RobotMoneyVault.sol` does none of this — there is no governance hook, no
   bucket structure, no vote integration, no basket leg. The roadmap doc
   (`docs/project-roadmap.md:69-79`) admits this, but the PRD reads as an
   "atemporal" spec of features that simply do not exist in the contract.

3. **`architecture-proposal-v0.md` §37 (dead-man's-switch) silently changes
   topic.** Sections 1–36 describe an agent payment gateway. Section 37
   pivots without transition to a *deposit-revert* primitive layered on an
   ERC-4626/4337 smart account, which is closer to (but still not aligned
   with) the deployed vault. The doc does not explain whether §37 replaces,
   complements, or is independent of §1–§36. It needs a relationship
   diagram or to be split into two documents.

4. **Several "must" requirements in the proposal are unimplementable as
   written or contradict each other.** Examples: gateway must verify
   `transferFrom moved exact amount` (§25, §33.2) but Section 37 explicitly
   uses `receive()` and per-deposit accounting that cannot work for
   ERC-20s without a wrapper — which Section 37 itself admits in §37.4.1
   and then ignores. The "client must verify gateway runtime code hash"
   (§28 step 8) sits next to "may be upgradeable behind a multisig+timelock
   proxy" (§26) — code hash pinning and proxy upgradeability are mutually
   incompatible without a code-hash-rotation protocol that is not
   specified.

5. **There is no explicit threat model for the *deployed* product.** The
   only document that discusses threats in any depth (the architecture
   proposal §1) is for an unrelated system. `docs/technical/security-review.md`
   is a finding-by-finding triage, not a threat model. The PRD's §7 is one
   page of bulleted reassurances. There is no document that says, for the
   actual ERC-4626 vault: "these adversaries exist, these are their
   capabilities, here is what each contract surface guarantees against
   each."

## 2. Internal inconsistencies and contradictions

### 2.1 Within `architecture-proposal-v0.md`

- **Upgrade vs code-hash pinning.** §26 permits proxy upgradeability behind
  a multisig+timelock; §5 (`gateway_code_hash`), §7 config example
  (`gateway_code_hash = "0xExpectedRuntimeCodeHash..."`,
  `architecture-proposal-v0.md:282`) and §28 step 8 mandate the client
  verify a runtime code hash before signing. Behind a proxy the runtime
  code hash is the *proxy's*, not the implementation's. The doc never
  resolves this. Either drop hash-pinning, or specify implementation-hash
  pinning + a rotation protocol where the client refuses signing during
  the timelock window and re-pins post-upgrade.

- **Refund destination vs threat model.** §1.3 lists "RPC responses unless
  cross-checked" and "client internal counters" as untrusted, yet the
  refund flow (§20.4) sends USDC to "payer/refund address defined at
  escrow time" — which is established by an agent-signed transaction
  whose `from` parameter the client controls. §37.4.2 admits this in the
  layered design ("original sender can itself be a contract") but the
  core spec does not handle it.

- **Window semantics.** §23.1 defines `windowId = block.timestamp /
  WINDOW_SECONDS`, marketed as "auditable." §23.3 then says "Threshold
  applies per fixed UTC-style on-chain epoch, not arbitrary rolling
  24-hour period." `block.timestamp / 86400` is *not* UTC midnight — it
  is the Unix epoch boundary, which happens to coincide with UTC
  midnight only because Unix epoch starts at 1970-01-01T00:00:00Z. This
  is fine, but the doc should not claim "UTC-style" without saying so;
  more importantly, §24 prescribes "gross spend counts toward threshold
  until window reset" with no mechanism to reset the WindowState struct
  defined at §17. Either each window is a fresh struct (memory model not
  shown) or rollover requires gas-paid maintenance not specified
  anywhere.

- **Agent vs daemon vocabulary.** The doc uses "agent," "daemon,"
  "client," and "agent key" interchangeably. §16 says "AGENT_ROLE - can
  create pending micropayments within policy"; §0 says "client" and
  "daemon"; §5.1 says "agent signing key." Are agent and daemon the
  same actor, or is the daemon a courier for the agent's key? §32 says
  "Daemon may submit confirmation tx but must not hold the master key" —
  fine, but earlier §1.1 trusts the master signer and §5 implies the
  daemon performs the agent signing. The role of the LLM ("Agent /
  Commerce Logic / LLM" in the §3 ASCII art) is unspecified beyond "high
  level payment request only." A clear actor list with capability
  matrices would resolve this.

- **`order_authorization` is undefined.** §8.2 includes
  `order_authorization: Vec<u8>` as a signed-tx input, and §20.1 includes
  `bytes calldata orderAuthorization`. The field is never specified —
  what signs it, what it binds to, why it exists, or how the gateway
  verifies it. If it is a merchant signature it changes the trust model
  significantly (merchants become trusted signers); if it is unused it is
  pure attack surface.

- **`paymentId` collisions.** §21 derives
  `paymentId = keccak256(chainid, address(this), msg.sender, orderId,
  merchantId, amount, nonceOrIdempotencyKey)`. The parameter is "nonce
  OR idempotency key" — these have opposite semantics (nonce is
  monotonic per-agent, idempotency key is caller-chosen). The contract
  cannot accept both interchangeably without changing collision
  guarantees. Pick one and define it.

### 2.2 Between proposal and PRD/roadmap

- **The PRD never references the proposal.** They describe different
  products. If the proposal is meant to specify a *future* payment layer
  on top of the vault, the PRD should at minimum cross-reference it under
  "Out of scope" or "Future work." Currently the only architecture-shaped
  doc in the repo describes a system the team is not building, and the
  product spec describes a system that does not match the contracts.

- **Governance cadence.** PRD §5.2 says weekly bucket-constituent votes
  and monthly weight votes. Roadmap §1 puts "$ROBOTMONEY weekly votes" in
  Phase 7+, "Future — not on-chain in current contracts." The PRD calls
  itself "atemporal" but reads as a current-state spec; it should mark
  every section that describes unimplemented behavior with a status
  badge or a "see roadmap §X" pointer, not a single disclaimer in the
  preamble.

### 2.3 Between PRD and the actual contracts

- **Adapter introspection.** PRD §5.1: "callers can enumerate active
  adapters and read per-adapter balances, target weights, and per-adapter
  caps." The contract exposes
  `adapters(uint256)` (auto-getter), `adapterCount()`,
  `getAdapterInfo(uint256)`, and `getAdapterDrift()`
  (`RobotMoneyVault.sol:506-541`). There is no enumerator that lists
  *only* active adapters — callers must read all and filter on
  `active`. Minor, but the PRD wording is more aspirational than the
  ABI.

- **"Atomic deposit-to-yield."** PRD §5.1 and the vault NatSpec
  (`RobotMoneyVault.sol:15-17`) both say "atomic deposit-to-yield." This
  is true *for the vault leg*, but PRD §5.2 also claims "the basket leg
  is a parallel routed swap committed atomically with the deposit." That
  is implemented (if at all) entirely in the CLI and is *not* atomic
  against the on-chain vault deposit — they are separate transactions.
  `docs/technical/smart-contracts.md:32` confirms "the vault knows
  nothing about the basket." The PRD wording will mislead an agent
  author into thinking the vault enforces this.

- **Caps enforced "atomically."** PRD §5.1 says "TVL cap and per-deposit
  cap, enforced atomically on `deposit`." This is true, but the
  security-review note that `totalAssets()` excludes idle USDC
  (`docs/technical/security-review.md:28-29`) means the TVL cap is *not*
  exact — it is approximated. The PRD should say "approximately
  enforced," because an integrator who reads "atomically enforced" and
  builds a cap-tight integration will be wrong by up to the unswept idle
  balance.

- **Three-fee model.** PRD §5.4 names management fee, swap-fee share,
  and exit fee as protocol parameters. Per
  `docs/technical/smart-contracts.md:147-148` and
  `RobotMoneyVault.sol:53,466-471`, only the exit fee exists on-chain.
  PRD does not flag this; project-roadmap §4 does. The PRD should say
  "exit fee on-chain; management and swap-fee-share off-chain pending
  Phase X." As written, it is misleading.

- **Synchronous redemption claim.** PRD §5.1: "no cooldown, no two-step
  claim, no unbonding window." The vault is honest about this for the
  USDC leg, but withdrawals depend on adapter liquidity. If Aave or
  Compound utilization is high, `_pullProportional` can return less than
  requested; the function caps `assetsNeeded` at total adapter balance
  (`RobotMoneyVault.sol:263`) but does not cap *per-adapter* shortfalls
  the way `AaveV3Adapter.withdraw` does
  (`AaveV3Adapter.sol:54-56` reverts with `WithdrawShortfall`). The
  PRD's claim of synchronous redemption is conditional on adapter
  liquidity and should be qualified.

- **`shutdown` semantics.** PRD §5.1 calls shutdown "permanent, terminal
  kill that disables deposits while preserving withdrawals." The
  implementation is consistent
  (`RobotMoneyVault.sol:155,446-450`), but PRD §7 ("emergency switches")
  does not say `shutdown` zeros `tvlCap` as a side effect, and does not
  warn that shutdown is unilateral by `EMERGENCY_ROLE` (no timelock, no
  multi-step). For an "atemporal product spec," irreversible admin
  powers belong in the trust model, not in the appendix.

## 3. Gaps and missing concerns

### 3.1 Security gaps

- **No documented adversary model for the deployed vault.** The proposal
  has §1.1–§1.4 (excellent for that other system); the PRD has §7 (one
  page, mostly reassuring). What is missing for the actual product:
  - What happens if Morpho/Aave/Compound is exploited? `forceRemoveAdapter`
    accepts the loss; shareholders bear it pro-rata. Is this acceptable?
    Documented? Pre-disclosed to depositors?
  - What is the trust assumption on the Safe multisig
    (`0x88bA…75A0`)? Number of signers, key custody, social-recovery,
    geographic distribution, signing latency in an emergency. None of
    this appears in `docs/`.
  - The `setExitFeeBps` ceiling is 1% but the floor is 0; admin can
    front-run a withdraw spike with a fee change in the same block.
    There is no timelock on fee changes. This deserves a paragraph in
    the trust doc.
  - `setFeeRecipient` is unilaterally settable
    (`RobotMoneyVault.sol:473-478`). A compromised admin key can
    redirect all *future* exit fees to an attacker. There is no warning
    or timelock. Mention this.

- **Adapter receipt-token assumptions.** The adapter contracts assume
  that `MORPHO_VAULT.convertToAssets(MORPHO_VAULT.balanceOf(this))` is a
  faithful live valuation (`MorphoAdapter.sol:50-53`), `aToken.balanceOf`
  is a faithful underlying (`AaveV3Adapter.sol:60-62`), and
  `COMET.balanceOf` is a faithful underlying
  (`CompoundV3Adapter.sol:66-68`). These are reasonable but unstated
  assumptions. If Morpho's `convertToAssets` were ever
  manipulable (e.g., via donation attacks during a bank run), the
  vault's `totalAssets()` becomes manipulable, which directly affects
  share price for all depositors. Document the dependency.

- **Allowance hygiene.** All three adapters use
  `safeIncreaseAllowance` then `forceApprove(0)` after deploy. This is
  correct, but `MorphoAdapter.deploy` performs the ERC-4626 deposit
  first and then clears the residual — if the underlying vault deposits
  *less* than the allowance for any reason, residual allowance lingers
  for one block until clearance. Not exploitable in practice (caller is
  `onlyVault`), but the convention should be documented as deliberate.

- **No incident-response runbook.** What does
  `EMERGENCY_ROLE` do at 3am if Aave is being drained? Who calls
  `pause()`? Is there an automated monitor? How is `emergencyWithdraw`
  decided? The trust model claims a Safe multisig but a multisig at
  3am with five geographically distributed signers is not an emergency
  response. No SLO, no on-call rotation, no documented monitoring
  pipeline. This belongs in `docs/technical/` somewhere.

### 3.2 Operational / failure-mode gaps

- **No documented rebalance policy.** `KEEPER_ROLE` is "intentionally
  not granted at launch" (`RobotMoneyVault.sol:129`). When and how does
  it get granted? To what infrastructure? The roadmap doesn't say. The
  vault accumulates drift; without a keeper, only ADMIN can call
  `rebalance()` or `adminRebalance()`. The system silently relies on
  manual ops.

- **`emergencyWithdraw` leaves USDC in the vault, not in users' hands.**
  After emergency withdraw the vault holds idle USDC and is paused —
  so users *cannot* redeem. Unpause requires `EMERGENCY_ROLE`. The
  irreversibility chain (pause → emergencyWithdraw → must unpause to
  refund users) is not documented anywhere in `docs/`. A user reading
  the PRD's "synchronous redemption" claim and seeing the vault paused
  has no doc to consult.

- **`forceRemoveAdapter` and accounting.** When EMERGENCY removes a
  loss-bearing adapter, `totalAssets()` *drops* immediately and so does
  the share price. There is no socialized-loss event; depositors who
  redeem in the same block before vs after the call get materially
  different outcomes. This is a fairness/MEV concern that deserves a
  paragraph in the trust model.

- **No description of upgradeability story.** The contract is
  non-upgradeable. If a bug is found, the only path is migrate-and-shut.
  The migration playbook (deploy v2, governance/admin moves liquidity,
  shutdown v1) is not described in any doc.

### 3.3 Testability gaps

- The architecture proposal's §34 enumerates Rust + Solidity test
  classes for a system that does not exist. There is no equivalent
  list for the actual vault.
- `docs/technical/test-framework.md` (referenced in `wc -l` output, 144
  lines) was not opened above; if it covers vault tests it should be
  cross-linked from the PRD's NFR §6 ("Test pyramid"). Currently the
  PRD describes a test pyramid in the abstract.
- Invariants worth testing on the actual contracts that no doc lists:
  - sum of `adapter.totalAssets()` matches `totalAssets()` exactly;
  - exit-fee accounting: `gross == net + fee` for all share/asset
    splits;
  - `_pullProportional` rounding remainder never exceeds
    `activeAdapterCount` wei;
  - `rebalance()` never moves more than `maxRebalanceBpsPerCall` × TVL.

### 3.4 Upgrade-path gaps

- The proposal is explicit (§26) about preferring non-upgradeable; the
  vault honors this. But the PRD describes governance-driven changes
  (weights, constituents, fees) that the *contract* does not support.
  The path from "PRD as written" to "PRD as deployed" is not in any
  doc. This is the core bookkeeping problem with calling the PRD
  "atemporal."

## 4. Unjustified or vague design decisions

- **Equal-weight by construction (`RobotMoneyVault.sol:490-493`).**
  `targetBps = MAX_BPS / activeAdapterCount`. This is hardcoded; admin
  cannot tilt without `adminRebalance(uint256[])`. The PRD §5.2 says
  weights are governance-set; the contract says weights are mechanically
  equal. No doc justifies the equal-weight choice or explains how the
  contract will graduate to governance-set weights without redeploy.

- **`MAX_ADAPTERS = 20` (`RobotMoneyVault.sol:35`).** Why 20? With
  `_routeDeposit` running two passes and `totalAssets()` looping over
  all adapters, 20 active adapters means a deposit calls
  `adapter.totalAssets()` upwards of 60 times. No gas analysis is
  documented. At 20 the per-deposit cost could be hundreds of
  thousands of gas before the user's deposit lands — a sustained DoS
  vector if an admin ever maxed out adapters.

- **Why exit fee, not entry fee?** The PRD §5.4 lists exit fee as the
  on-chain fee but does not justify the choice. Entry fees deter
  inflation attacks; exit fees deter exit. Designers usually pick one
  for a reason — say which.

- **`MAX_REBALANCE_BPS_CEILING = 5000`** (50%). Why 50? Why not 25? The
  comment at `RobotMoneyVault.sol:37` says "keeper can never move >50%
  per call" but the *initial* value is 25%. The 50% ceiling means a
  compromised ADMIN can set the keeper limit higher than the operator
  comfort level without any timelock. Justify or lower.

- **`MIN_REBALANCE_INTERVAL_FLOOR = 1 hours`.** Floor of 1 hour means a
  keeper can be configured to rebalance hourly — fine for ops, but if
  rebalance gas costs are non-trivial relative to APY differential the
  protocol could burn yield on rebalances. No doc on rebalance economics.

- **§37.4.3 "B pre-authorizes a list of allowlisted senders whose
  deposits auto-confirm."** This re-introduces the trust assumption the
  rest of §37 was trying to remove. If allowlist write authority is
  held by B (offline hardware) it is fine; if it is held by H, the
  layered design collapses. Section 37 does not say.

- **§37.4.4 "small bot, or you accept...sit in pending forever."** This
  is a real UX cliff for low-value deposits. The doc names Gelato /
  Chainlink Automation as solutions but does not commit to one or
  enumerate the trust dependencies that adds (Gelato can censor
  reverts, for example).

## 6. Concrete suggestions for improvement

1. **Rename and re-scope `architecture-proposal-v0.md`.** Either:
   (a) split into `architecture-payment-gateway-proposal.md` (sections
   1–36) and `architecture-deposit-clawback-proposal.md` (section 37),
   each clearly marked "design proposal — not deployed";
   (b) move both under `docs/proposals/` so they don't sit next to
   current-state docs in `docs/`;
   (c) delete the v0 if it has been superseded.

2. **Add `docs/technical/threat-model.md`.** Explicitly for the deployed
   vault. Cover: adversaries (compromised admin, compromised keeper,
   malicious adapter, malicious underlying protocol, malicious LP/MEV,
   malicious depositor, USDC issuer freeze), capabilities of each, and
   what the contract guarantees against each. Cross-reference the
   security review.

3. **Add status badges to the PRD.** Every functional-requirement
   subsection should carry a `[Phase 4 — implemented]`, `[Phase 5 —
   in progress]`, or `[Phase 7+ — future]` tag. The "atemporal" framing
   is fine for an aspirational doc but only if it does not lie about
   present-day capability.

4. **Resolve the governance/admin equivocation.** Pick one of:
   - "All governance is the Safe multisig today; on-chain voting is
     future work" — and say this on every PRD §5 page that mentions
     governance;
   - or build the governance integration before the PRD claims it.

5. **Document the 2% management fee mechanic.** If it is off-chain
   transfers from `feeRecipient`, say so in PRD §5.4 with a citation
   to the operational policy. If it is "we will add a fee accrual to
   the vault later," add it to the roadmap with a target phase.

6. **Document the migration story.** A single-page
   `docs/technical/upgrade-and-migration.md`: how the team would
   migrate from v1 to v2 if a bug were found; how they would roll out
   governance; how `KEEPER_ROLE` will be granted; what triggers
   `shutdownVault()` and what comes after.

7. **Strengthen the proposal's spec or remove "MUST".** §11
   ("client must enforce…") and §33 ("revert if…") read as a
   requirements doc but several items are unimplementable as written
   (code-hash pinning behind a proxy, exact-amount transferFrom for a
   token that is not fee-on-transfer). Either soften to "should" with
   rationales, or split implementation-feasible from aspirational
   constraints.

8. **Tighten the security review.** `docs/technical/security-review.md`
   filters out the `totalAssets()`-excludes-idle-USDC issue at
   confidence 2 because "no theft path." Fine as a vulnerability
   verdict, but the *invariant violation* is real and should be
   logged as an accounting bug to fix in a future deployment, not
   buried in a "near-threshold" footnote.

9. **Justify the equal-weight rebalance design.** A short paragraph
   in `docs/technical/smart-contracts.md` §3.5 explaining why equal
   weight (vs. APY-tilted) was chosen, and what would have to change
   to support governance-set tilts.

10. **Cross-link the docs.** PRD does not mention
    `architecture-proposal-v0.md`. The proposal does not mention the
    PRD or the vault. `project-roadmap.md` mentions both but not the
    threat model or the security review. Add a `## References`
    section to each top-level doc with the doc graph.
